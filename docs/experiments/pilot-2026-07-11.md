# Cost-efficiency pilot — 2026-07-11

Format: see `docs/experiments/README.md`. Fixture: `eval/fixtures/toy-lib-template/`
scaffolded via `eval/harness/scaffold.sh` with personas `planner,reviewer`
(+ core `orchestrator`/`explorer`/`lead-programmer`). Task: `eval/tasks/feature-task.md`,
scored against `npm test` plus the held-out invariant test in
`eval/tasks/feature-task.holdout.test.js` (independent defect-catch gate —
the fixture's `applyDiscountPercent` mutates its input in place; a naive
`previewDiscountedTotal` that reuses it silently corrupts the cart).

## Baseline

- **Variant**: `baseline` (unmodified `agents/*.md`/`hooks/scripts/*.sh` as
  scaffolded, no overlay applied).
- **Reps**: 5 (one harness-validation dry run + 2 from matrix attempt #1 +
  2 from matrix attempt #2 — all five are the same unmodified condition, so
  pooled here).
- **Result**: all 5 succeeded end-to-end. `tests_pass_own`: 5/5.
  `tests_pass_with_holdout`: 5/5 — the pipeline correctly avoided the
  planted mutation trap (copied before preview) every time, independently
  confirmed by the harness's held-out test, not just its own claim.
  - cost (USD): 0.647, 0.690, 0.516, 0.616, 0.735 — mean **$0.641**
  - turns: 7, 12, 8, 7, 8 — mean **8.4**
  - wall-clock (s): 193, 187, 141, 171, 199 — mean **178s**
  - Raw rows: `eval/results.jsonl`, variant=`baseline`.
  - Note the spread (turns 7-12, cost $0.52-$0.74) — this is the noise floor
    every variant result below has to clear before a delta means anything.

## Variant matrix attempt #1 — invalidated, not a real result

All 10 variant runs (5 hypotheses × 2 reps) failed within 3-5s at `exit=1`,
`total_cost_usd=0`, `num_turns=1` — **not** a finding about any hypothesis.
Root cause, confirmed via `.eval-stderr.log`/`.eval-result.json`: the
account hit a session-level rate limit (`api_error_status: 429`,
`"You've hit your session limit"`) immediately after the baseline reps
consumed the available budget, so every subsequent `claude -p` call failed
before making any real API request. A second, independent bug was also
found and fixed regardless: each new scratch fixture dir is an untrusted
Claude Code workspace, so `settings.json`'s `permissions.allow` was being
silently ignored ("this workspace has not been trusted") — `scaffold.sh`
now pre-trusts each scratch dir via `~/.claude.json`
(`projects.<path>.hasTrustDialogAccepted = true`), and
`eval/harness/cleanup.sh` removes that entry again when a scratch dir is
torn down. These 10 invalid rows were not left in `eval/results.jsonl`'s
interpretation below — retry is pending session-limit reset.

## Candidate hypotheses (finalize exact set at kickoff)

1. **lead-programmer maxTurns cap** — never added per CHANGELOG 0.3.0, the
   system's highest-frequency persona. Prediction: turns/cost drop with no
   effect on `tests_pass_own`/`tests_pass_with_holdout`.
2. **Terse reviewer verdict-only output contract** — the post's
   highest-value lever. Prediction: reviewer-attributable cost drops, paired
   rerun checks `tests_pass_with_holdout` doesn't regress (the post's own
   named failure mode: terseness silently dropping catch rate).
3. **Trim reviewer.md's explanatory HTML-comment preamble** — confirm
   empirically whether it's sent to the model at all before assuming any
   savings; if it isn't, this hypothesis is void and should be logged as
   such rather than run.
4. **Pre-baked diff/changed-files packet for the reviewer** instead of ad
   hoc git discovery — no existing script does this today (confirmed via
   repo grep during planning).
5. **Tighten explorer maxTurns** below its current 10.

## Variant matrix attempt #2 — real results

Rate limit cleared earlier than the stated 2:40pm estimate (confirmed via a
one-off probe run before committing to the full matrix); `pilot.sh` then
completed all 10 variant runs successfully. **N=2 per variant is small** —
treat single-digit-percent deltas as noise given the baseline's own spread
(above); only deltas clearly outside that spread are read as signal below.

### E1: lead-programmer maxTurns cap

- **Hypothesis**: adding `maxTurns: 30` to `lead-programmer` (never set per
  CHANGELOG 0.3.0) reduces turns/cost with no quality effect.
- **Prediction**: turns/cost drop, `tests_pass_with_holdout` stays at 100%.
- **Change**: `eval/variants/lead-programmer-maxturns/agents/lead-programmer.md`
  — added `maxTurns: 30` to frontmatter only.
- **Result (N=2, initial)**: cost $0.604 (-5.8%), turns 4.0 (-52.4%), wall
  170s (-4.6%), holdout 2/2.
- **Result (N=5, topped up)**: costs $0.645, 0.563, 0.665, 0.489, 0.511 —
  mean **$0.574 (-10.4%)**, turns mean **5.2 (-38.1%)**, wall mean **151s
  (-15.4%)**, `tests_pass_with_holdout` **5/5**. Every one of the 5 reps
  beat baseline's mean cost; direction held consistently, not just on the
  first 2.
- **Verdict**: CONFIRMED. Consistent, meaningful improvement across all 3
  metrics with 5 independent reps and zero quality loss.
- **Decision**: SHIP candidate.

### E2: terse reviewer verdict-only output contract

- **Hypothesis**: a strict "verdict-first, no restated context" contract on
  `reviewer`'s final message (the post's highest-value lever) cuts cost
  without dropping defect-catch rate.
- **Prediction**: reviewer-attributable cost drops; `tests_pass_with_holdout`
  stays at 100% (the post's own named failure mode is terseness silently
  dropping catch rate — this is the check for that).
- **Change**: `eval/variants/terse-reviewer/agents/reviewer.md` — rewrote the
  "Verdict" bullet into an explicit terse, verdict-only contract.
- **Result (N=2, initial)**: cost $0.521 (-18.6%), turns 7.5 (-10.7%), wall
  138s (-22.5%), holdout 2/2.
- **Result (N=5, topped up)**: costs $0.535, 0.507, 0.490, 0.595, 0.509 —
  mean **$0.527 (-17.7%)**, turns mean **4.8 (-42.9%)**, wall mean **142s
  (-20.1%)**, `tests_pass_with_holdout` **5/5**. Tightest, most consistent
  cluster of any variant tested (range $0.49-$0.595 vs. baseline's much
  wider $0.516-$0.735) — low variance as well as a large mean effect.
- **Verdict**: CONFIRMED — the strongest, cleanest result in this pilot.
  Effect size held almost exactly from N=2 to N=5 (-18.6% → -17.7% on cost),
  which is itself a good sign this isn't noise.
- **Decision**: SHIP. Best cost/quality tradeoff found; recommend applying
  to the real `agents/reviewer.md` next.

### E3: trim reviewer.md's HTML-comment preamble

- **Hypothesis**: the ~10-line explanatory HTML comment in `reviewer.md`
  (design rationale for humans) is sent to the model as part of its system
  prompt and costs tokens with no runtime value.
- **Prediction**: if the comment is actually stripped before reaching the
  model, this hypothesis is void (no measurable effect); if it's sent
  verbatim, cost/turns should drop.
- **Change**: `eval/variants/trim-reviewer-comment/agents/reviewer.md` —
  deleted the comment block entirely, rest byte-identical to baseline.
- **Result**: cost $0.585 (-8.8%), turns 3.5 (**-58.3%**, though rep1's
  turns=1 is an outlier pulling this down), wall 162s (-8.8%),
  `tests_pass_with_holdout` 2/2.
- **Verdict**: PLAUSIBLE the comment is sent to the model (a null hypothesis
  would predict ~0% delta everywhere, and cost/wall both moved in the same
  direction as E1/E2) — but turns=1 on rep1 is suspicious enough (given no
  other variant or baseline rep ever hit 1) that this needs a rerun before
  trusting the magnitude.
- **Decision**: NEEDS MORE DATA — re-run before deciding SHIP/REJECT; rep1's
  outlier turn count could be a scoring artifact, not a real effect.

### E4: pre-baked review packet instead of ad hoc git discovery

- **Hypothesis**: having the orchestrator pre-build a diff/log packet before
  spawning the reviewer (so the reviewer reads one file instead of running
  its own `git diff`/`git log`) cuts reviewer-side discovery cost.
- **Prediction**: cost/turns drop for reviewer-touching work.
- **Change**: `eval/variants/review-packet/agents/orchestrator.md` (added a
  packet-write step to review routing) +
  `eval/variants/review-packet/agents/reviewer.md` (reads the packet first,
  falls back to git only if missing/stale).
- **Result**: cost $0.967 (**+50.9%**), turns 9.5 (+13.1%), wall 271s
  (**+52.3%**). rep1 actually hit the harness's own `--max-budget-usd 1.00`
  circuit breaker and aborted (`error_max_budget_usd`, cost $1.10) before
  finishing; rep2 completed at $0.84 with 18 turns. `tests_pass_with_holdout`
  2/2 on the reps that did finish.
- **Verdict**: REFUTED — clearly worse, not better, and not just noise (one
  rep didn't even finish inside budget). Root cause is visible in the
  mechanism, not mysterious: this fixture's diff is trivially small, so the
  orchestrator's extra `git diff`/`git log`/file-write overhead exceeds
  whatever discovery cost it was meant to save the reviewer. The premise
  (ad hoc git discovery is expensive) may still be true on a real, larger
  diff — this fixture is the wrong size to test it.
- **Decision**: REJECT for this fixture. Not a verdict on the underlying
  idea — would need a fixture with a genuinely large diff to test fairly.

### E5: tighten explorer maxTurns (10 → 6)

- **Hypothesis**: explorer's current `maxTurns: 10` has headroom to spare on
  simple structural lookups; tightening it saves cost with no quality loss.
- **Prediction**: cost/turns drop, no quality effect (explorer never touches
  the acceptance criteria directly).
- **Change**: `eval/variants/explorer-maxturns-tight/agents/explorer.md` —
  `maxTurns: 10` → `maxTurns: 6`, nothing else changed.
- **Result**: cost $0.681 (+6.3%), turns 6.0 (-28.6%), wall 207s (+16.5%).
  High spread between the two reps (cost $0.871 vs. $0.491 — nearly 2x),
  `tests_pass_with_holdout` 2/2.
- **Verdict**: INCONCLUSIVE — no consistent direction, and the two reps
  disagree with each other more than either disagrees with baseline. Cost
  went the wrong way on average.
- **Decision**: REJECT as tested. The high variance suggests explorer's
  turn count isn't the fixture's actual cost driver either way — not worth
  more reps on this fixture; would need a task that actually stresses
  explorer's blast-radius work to test this fairly.

## Summary

After topping up E1 and E2 to N=5 (matching baseline's sample size): **2
confirmed SHIP candidates** (E2 terse reviewer contract: cost -17.7%, wall
-20.1%, turns -42.9%; E1 lead-programmer maxTurns cap: cost -10.4%, wall
-15.4%, turns -38.1% — both held consistent from N=2 to N=5), **1 needs
more data** (E3 trim comment — plausible but one outlier rep still
unresolved), **2 rejected** (E4 review packet — wrong-sized fixture to test
it fairly; E5 explorer maxTurns — no signal, high variance). Quality held on
every single run across all 18 total (12 initial + 6 top-up):
`tests_pass_with_holdout` was 18/18 — no variant ever silently dropped the
planted-defect catch, including the terse-reviewer contract that the source
post specifically flagged as a risk for that exact failure mode.

**Combining E1+E2** (both touch different personas — lead-programmer and
reviewer — so plausibly stack) is a natural next experiment before shipping
either individually, but wasn't run in this pilot.
