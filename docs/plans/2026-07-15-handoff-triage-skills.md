# Plan: wire the `handoff` + fail-scoped `triage` skills (2026-07-15)

**Issue-retrieval contract (per shared protocol):** issues live in GitHub
issues (repo `Storreslara/AntiSlop`, `gh` CLI authenticated). File each unit
with `gh issue create --label plan/2026-07-15-handoff-triage-skills`; fetch
with `gh issue list --label plan/2026-07-15-handoff-triage-skills`. (Source:
`.claude/persona-config.json` `issueTracker`.) Slicing this plan into
per-unit issues is `task-master`'s `to-issues` job, done ONLY after this plan
is approved and the single Open Question below is resolved — `spec-master`
does not slice.

## Goal
Wire two `mattpocock/skills` into the antislop persona system as new, small,
loosely-coupled additions on top of the completed 2026-07-14 threefold
update:
1. **`handoff`** — a near-verbatim passthrough of the published productivity
   skill, wired as a **7th `<MATTPOCOCK:slot>` substitution**
   (`<MATTPOCOCK:handoff>`), preloaded into `lead-programmer` only, and made
   available project-wide as the `/handoff` slash command. It compacts a
   conversation into a handoff doc (OS temp dir) so a fresh session can pick
   up a cut-off unit of work.
2. **`fail-triage`** — a **first-party TAILORED derivative** of the published
   `triage` skill (NOT a passthrough), scoped **only to the post-FAIL path**
   per the user's exact non-negotiable "upon fail" constraint. It ships via
   the plugin-source `skills/` + `plugin.json` path like `pathfinder`, drops
   `triage`'s transitive deps and its entire issue/PR state machine, and
   sharpens `spec-master`'s existing debug-spec step with an explicit
   verify/reproduce + categorize front-half.

Explicit non-goals (scope discipline, per the user's "keep it small" and
constitution): NO general issue/PR triage, NO triage state machine / labels /
PR request-surface, NONE of mattpocock's other ~9 engineering skills, NO
`handoff` tailoring beyond the slot, NO adapter (cursor/codex) ports this
release, NO change to the reviewer PASS gate or the normal first-FAIL routing.

## Context
This repo IS the antislop plugin source AND self-hosts it (dogfoods), so most
changes touch **two copies** of a persona file: the plugin source
`agents/*.md` (carries `<MATTPOCOCK:slot>` placeholders) and the ADAPT output
`.claude/agents/*.md` (carries resolved skill names, regenerated by
`node bin/cli.js --update` via `applyMattpocockSubs`). The threefold update
that just shipped established every mechanism this plan reuses:
- **Passthrough mattpocock slot** (threefold Step 2.1, `to-spec`): register
  `"<slot>": "<installed-name>"` in `.claude/persona-config.json`
  `substitutions.mattpocockSkills`, add the purpose to `install-antislop`
  step 3's by-purpose TUI list, add a README credit, and place the
  `<MATTPOCOCK:slot>` in the consuming persona's `skills:` frontmatter. There
  are currently 6 slots (`grill-me`, `to-issues`→`to-tickets`,
  `improve-codebase-architecture`, `tdd`, `diagnose`→`diagnosing-bugs`,
  `to-spec`); `handoff` becomes the 7th.
- **First-party tailored skill** (threefold Step 2.3, `pathfinder`): author
  `skills/<name>/SKILL.md` under the plugin SOURCE `skills/` dir, carry a
  `Derived from mattpocock/skills <x>` provenance note, drop the upstream
  skill's transitive deps, and reference it as `antislop:<name>` in the
  consuming persona's `skills:` line (same string in both copies — no
  substitution). `fail-triage` follows this exactly.

**Current version state (load-bearing for Constitution P3):** `plugin.json`
and `package.json` both read `0.9.0`, and the new persona files still carry
`v0.9.0` stamps — i.e. the threefold update's own `0.10.0` version-bump
consolidation had not yet stamped at the time this plan was authored. See
Risks for how this plan's version bump coordinates with it.

**Why these two skills fit real, evidenced needs:**
- `handoff`: `lead-programmer`'s multi-turn resumable build loops are the
  system's one documented multi-resume pain point (several dispatches were
  cut off mid-turn this session and needed ad-hoc manual resume-with-context
  messages). The shared protocol's "Continuing after a FAIL verdict" section
  already prescribes bundling "a self-contained prompt with the original plan
  step, a one-line diff summary, and the defect list" — `handoff` formalizes
  exactly that artifact. It **complements, never replaces**, the existing
  continuity mechanisms (see the Risk on non-duplication).
- `fail-triage`: this project already routes FAILs (normal FAIL → defect list
  to `lead-programmer`; 2-FAIL cap → `spec-master` debug spec →
  `task-master` re-derive). The debug-spec step currently reasons from the
  `.fail` text + fix-attempt commits but does not mandate a fresh
  reproduce/categorize front-half — which is precisely `triage`'s
  verify-then-categorize contribution, one level up (diagnosing the SPEC, not
  the code).

## Risks / dependencies
- **Version-bump coordination (Constitution P3):** this plan targets a
  `0.11.0` bump *assuming* the threefold update's own `0.10.0` consolidation
  lands first. If, at execution time, `plugin.json` still reads `0.9.0`
  (threefold's bump not yet stamped), Step C.1 bumps to the next minor above
  whatever is current and the CHANGELOG must not skip a version — the
  acceptance criteria below check "strictly greater than the pre-edit version
  and package.json matches", which is robust to the exact number. Do NOT run
  this plan's C.1 concurrently with threefold's version bump.
- **`handoff` is `disable-model-invocation: true`** (published frontmatter):
  the persona will not auto-select it from its description. Preloading it in
  `lead-programmer`'s `skills:` makes the persona AWARE of it and able to
  invoke it explicitly via the `Skill` tool (or the human via `/handoff`).
  This is the intended behavior — do not strip the flag when substituting the
  slot (passthrough = verbatim).
- **`handoff` must not duplicate existing continuity machinery:** the
  `lead-programmer` body note (Step A.2) must state that `handoff` complements
  — never replaces — the WIP sentinel (mechanical pause signal) and does not
  change any gate. A handoff doc is a resumption aid; the sentinel is a
  turn-end permission. Conflating them would risk agents writing rich docs to
  dodge a red suite (the exact misuse the WIP-audit log guards against).
- **`fail-triage` overlaps `spec-master`'s debug spec (be honest about it):**
  under the recommended default (OQ1), `fail-triage`'s net-new value over the
  existing debug-spec prose is only the explicit **verify/reproduce +
  categorize** front-half. If the user judges that not worth a separate skill,
  a lighter alternative is to fold that front-half into the debug-spec section
  prose with NO new skill — noted here, but this plan delivers the skill as
  the user asked. `fail-triage` deliberately does NOT carry `triage`'s
  agent-brief format: the debug-spec's existing revised-step format IS the
  brief.
- **Track B's blast radius is fixed (OQ1 resolved to option (a),
  2026-07-15):** Track B touches NO routing — `orchestrator.md` and
  `persona-protocol.md` are UNCHANGED; `fail-triage` is invoked only from
  within `spec-master`'s existing debug-spec step. The every-FAIL
  early-routing-gate variant is dropped from scope and must not be
  implemented.
- **`spec-master` is an OPTIONAL persona (Constitution P4):** wiring
  `fail-triage` into `spec-master` must degrade gracefully — a project that
  skips `spec-master` simply never invokes the skill; the skill ships unused,
  breaking nothing. `lead-programmer` is CORE, so `handoff`'s preload carries
  no optional-persona risk.
- **Track independence:** Track A (`handoff`) and Track B (`fail-triage`)
  touch disjoint files and can proceed in parallel; both precede Track C
  (version bump + CHANGELOG consolidation), which must run last.
- **Deterministic-config discipline (Constitution P2):** the
  `mattpocockSkills` map entry and any `.claude/agents/*.md` regeneration go
  through `node bin/cli.js --update` / `applyMattpocockSubs`, never by
  hand-editing the resolved adapted copy or `fileHashes`.
