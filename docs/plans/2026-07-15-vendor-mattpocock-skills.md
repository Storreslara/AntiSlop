# Plan: vendor the mattpocock/skills the agent team uses, and delete the install/substitution machinery (2026-07-15)

**Issue-retrieval contract (per shared protocol):** issues live in GitHub
issues (repo `Storreslara/AntiSlop`, `gh` CLI authenticated). File each unit
with `gh issue create --label plan/2026-07-15-vendor-mattpocock-skills`; fetch
with `gh issue list --label plan/2026-07-15-vendor-mattpocock-skills`. (Source:
`.claude/persona-config.json` `issueTracker`.) Slicing this plan into per-unit
issues is `task-master`'s `to-issues` job, done ONLY after this plan is
approved — `spec-master` does not slice.

## Goal
Replace the `<MATTPOCOCK:slot>` substitution mechanism entirely with the
plugin-source first-party pattern already used for `pathfinder`/`fail-triage`:
**vendor** the mattpocock/skills content the agent team uses directly into this
repo's `skills/` tree (with credit), ship it via the normal plugin path, and
**delete** the external-install dependency and the substitution machinery so a
consumer needs no `npx skills@latest add mattpocock/skills` step when adopting
antislop. Per the operator's four resolved design decisions (2026-07-15), this
is the **large/comprehensive** end of the scope spectrum:

1. **OQ1 → remove the machinery entirely.** Delete `MATTPOCOCK_RE`,
   `applyMattpocockSubs`, `deriveMattpocockSubsForFile`, `hasMattpocockResidue`,
   the `substitutions.mattpocockSkills` map, the `--with-mattpocock`/
   `--only-mattpocock` install path, the install-antislop mattpocock-selection
   section, and the related tests.
2. **OQ2 → full-fidelity vendoring of the whole dependency tree**, not tailored
   derivatives. Verbatim copies (+ provenance/MIT notice) of every used skill
   AND its transitive skill dependencies.
3. **OQ3 → port the real `/grilling` content.** `grill-me` and `grilling` are
   distinct upstream skills; vendor both, so `grill-me` genuinely resolves.
4. **OQ4 → a documented, actionable periodic re-sync process** for drift.

## Context

This repo IS the antislop plugin source AND self-hosts it (dogfoods), so most
persona changes touch **two copies**: the plugin source `agents/*.md` and the
ADAPT output `.claude/agents/*.md` (regenerated by `node bin/cli.js --update`).
Two mechanisms currently ship mattpocock-derived content:
- **`<MATTPOCOCK:slot>` substitution** (being deleted): a placeholder in a
  persona `skills:` line resolves at ADAPT/`--update` time via
  `persona-config.json`'s `substitutions.mattpocockSkills` map to an
  EXTERNALLY-installed skill name. The skill CONTENT never lived in this repo.
- **First-party plugin-source skill** (the target pattern): content authored/
  vendored into `skills/<name>/`, registered in `.claude-plugin/plugin.json`,
  referenced as a plain `antislop:<name>` string, shipped to every consumer
  with no external install. `pathfinder` and `fail-triage` already work this
  way.

**Verified current wiring (grep of `agents/*.md`) — 7 wired slots:**
`grill-me` (spec-master, milestone-auditor), `to-issues`→`to-tickets`
(task-master), `improve-codebase-architecture` (scribe), `tdd`
(lead-programmer body), `diagnose`→`diagnosing-bugs` (lead-programmer body),
`to-spec` (spec-master), `handoff` (lead-programmer). `tdd`/`diagnose` are
body-referenced (invoked on demand via the `Skill` tool), not frontmatter-
preloaded, but still require their content on disk to be invokable.

**Full transitive dependency closure (fetched from mattpocock/skills `main`
and traced):** the 7 wired skills pull in 5 further skills. `grill-me`
(`productivity/grill-me`, body "Run a `/grilling` session") → `grilling`
(`productivity/grilling`, distinct, self-contained). `improve-codebase-

**Vendoring boundary (where it stops — closure is CLOSED, not unbounded):**
- **`setup-matt-pocock-skills` is NOT vendored.** It is a config wizard
  (issue-tracker/triage-label/doc-layout setup) that antislop already replaces
  with `install-antislop` + `persona-config.json` `issueTracker` + the
  retrieval contract. Vendoring it verbatim would ship a *second* setup
  process — the exact thing this goal ditches. References to it inside the 3
  skills that mention it (`to-spec`, `to-tickets`, `code-review`) are
  **repointed** to antislop's native mechanism (a minimal, documented,
  provenance-noted edit — see the fidelity Risk).
- **The `agents/openai.yaml` adapter files are NOT vendored** (antislop ships
  Claude-format skills only; matches the "no cursor/codex adapter ports"
  precedent from the 2026-07-15 handoff/triage plan).
- No skill outside this 12 is referenced by the closure. `/adr`, `/spec`,
  `/PRD`, `/hunk`, `/agents`, `/issue-tracker`, `CONTEXT-MAP.md` are section/
  argument/project-artifact references inside vendored skills, not external
  skills — verified against the full recursive upstream tree.

**License:** mattpocock/skills is **MIT** (© 2026 Matt Pocock). Verbatim
vendoring is permitted provided the copyright + permission notice is retained
"in all copies or substantial portions." This plan ships the full MIT text
once in a NOTICE file and a per-skill provenance line pointing to it.

**Current version state:** `plugin.json` and `package.json` both read
`0.11.0`. This plan targets `0.12.0` (see Risks for robustness to that).

## Risks / dependencies
- **Fidelity-vs-goal tension (be explicit):** OQ2 asks for *verbatim* copies,
  but the *goal* is to ditch the setup process. The 3 skills that reference
  `/setup-matt-pocock-skills` (`to-spec`, `to-tickets`, `code-review`) cannot
  be BOTH byte-verbatim AND free of the setup wizard. Resolution: they are
  vendored verbatim EXCEPT for a minimal, explicitly-recorded repoint of the
  `/setup-matt-pocock-skills` reference to antislop's native mechanism, noted
  in each of those three skills' provenance headers. The other 9 skills are
  truly byte-verbatim (provenance header aside). This is the only deviation
  from strict verbatim and it is goal-mandated.
- **`bin/cli.js` removal blast radius (Constitution P2 — the central risk):**
  the machinery is twice-hardened and its removal is a large diff:
  `MATTPOCOCK_RE` (l.202), `applyMattpocockSubs` (l.204), `deriveMattpocock-
  SubsForFile` (l.283), the backfill loop that populates `mattpocockSkills`
  (l.370-382), `hasMattpocockResidue` (l.434), the render call (l.497), the
  up-front residue detection (l.561-582), the `--with-mattpocock`/`--only-
  mattpocock` path (l.1502-1523), the `module.exports` entries (l.1579-1590),
  and the `--only-mattpocock` branch in `bin/install-deps.sh`. Removal MUST be
  TDD-first: delete/rewrite the mattpocock cases in `tests/cli-backfill.test.js`
  first, keep every non-mattpocock test (MCP wiring, `fileHashes`, backfill of
  OTHER fields) green. `node bin/cli.js --update` must still exit 0 on this
  repo after removal.
- **Ordering dependency:** the machinery can only be removed once NO source
  file carries a `<MATTPOCOCK:slot>` (else `--update` would need the
  substitution). Therefore Track B (repoint personas) MUST complete before
  Track C (remove machinery). Track A (vendor skills) MUST precede Track B (the
  `antislop:<name>` references must resolve to real shipped skills).
  **Intra-window test gap (added 2026-07-15, surfaced executing B.1/#51):**
  removing the slots in B.1 reddens one live-source subtest in
  `tests/cli-backfill.test.js` that C.1 is scoped to delete wholesale; nothing
  otherwise keeps `validate.sh` green in the B.1→C.1 window. Step B.1b (a narrow
  guard mirroring the file's existing `if (!sourceBody.includes('<MATTPOCOCK:'))
  return;` pattern) restores green — B.1b MUST land in the same change window as
  B.1, and B.1's own validate.sh gate is deferred exactly one step to B.1b.
- **Capability loss (document, don't silently drop — operator's OQ1 note):**
  removing the machinery removes the documented ability for a consumer to wire
  an arbitrary *unported* mattpocock skill via a `<MATTPOCOCK:slot>` + map
  entry. This is an intentional, documented loss — a consumer now adds any
  skill as a first-party `skills/<name>/` entry instead. Step F.1 records this
  in the CHANGELOG and a new ADR; Step D.1 removes the `update-fallback.md`
  "Case A — unresolved `<MATTPOCOCK:slot>`" recovery path that assumed it.
- **Issue-tracker config must survive (Step D.2):** `/setup-matt-pocock-skills`
  currently seeds `persona-config.json`'s `issueTracker`. Removing the
  mattpocock flow without a native replacement would break the retrieval
  contract task-master depends on. D.2 adds a native `install-antislop` capture
  step; its acceptance verifies `issueTracker` is still populated.
- **Version-stamp discipline (Constitution P3):** this plan edits many
  `agents/*.md` and templates; every edited version-stamped file must be
  re-stamped and `plugin.json`/`package.json` bumped with a CHANGELOG entry in
  the dedicated Step F.1. Do NOT run F.1's bump concurrently with any other
  in-flight version bump.
- **Optional-persona degradation (Constitution P4):** `spec-master`,
  `task-master`, `scribe`, `milestone-auditor` are optional; the vendored
  skills ship regardless, so a project skipping any of them simply never
  invokes the corresponding skill — nothing breaks.
- **Name-collision check:** the 12 upstream names do not collide with existing
  antislop skills (`coding-discipline`, `pathfinder`, `roast-work`,
  `fail-triage`). `fail-triage`/`pathfinder` remain tailored derivatives of
  `triage`/`wayfinder`, which are NOT in this closure — no conflict.
- **Deterministic-config discipline (Constitution P2):** all
  `.claude/agents/*.md` regeneration and `fileHashes` updates go through
  `node bin/cli.js --update`, never by hand-editing the adapted copies.
